# Digital Element - тестовое задание по вёрстке

Проект выполнен как решение тестового задания «Цифровой Элемент». Сборка, стили и код намеренно организованы так, чтобы продемонстрировать навык архитектурного подхода и модульности. Более подробно различные архитектурные решения описаны ниже, в том числе в последнем абзаце.

## Стек и принципы

- **Vite** + **TypeScript** (ES6+).
- **SCSS** (без UI-библиотек и сеток; только собственные стили).
- **ESLint** с правилами от «Цифрового Элемента» (пакет `@delement/eslint-config-master`).
- **SVG** для иконок и части графики.
- Без сторонних JS/TS-библиотек и фреймворков: функционал реализован на «чистом» TS.
- Разметка и стили в БЭМ-стиле, с упором на семантику и адаптивность.

## Команды

Требуемые команды доступны в `package.json`:

- `npm run dev` - запуск в режиме разработки.
- `npm run build` - production-сборка.
- `npm run lint` - линт исходников.
- `npm run preview` - предпросмотр production-сборки.

## Быстрый старт

Требования: Node.js 20+ и npm.

1. `npm install`
2. `npm run dev`
3. Открыть `http://localhost:3000`

Production-сборка: `npm run build`, результат будет в `dist/`.

## Структура проекта

- `src/`
  - `main.ts` - точка входа приложения.
  - `app/` - модульная логика (классы, разбиение по доменам: анимации, наблюдатели, формы, UI-компоненты).
  - `fonts/`- шрифты (правда с Galien были трудности, см. конец файла README)
  - `styles.scss` и `styles/` - SCSS-слои/части (шрифты, переменные, базовые стили, секции и компоненты).
- `public/` - статические файлы, которые копируются в корень `dist/` при сборке.
- `assets/` - исходные ассеты, используемые в проекте.
- `dist/` - файлы production-сборки (генерируются).
- `.github/workflows/` - CI/CD пайплайн.

## Анимации и оверхед архитектуры

В проекте намеренно реализовано больше анимаций и эффектов, даже если где-то это не соответствует лучшим практикам UI/UX и быстродействию. Это сделано для демонстрации навыков (для более крупного проекта я бы использовал gsap, возможно даже three.js. Также это можно оптимизировать, но вряд ли это требовалось от тестового)

Также в коде сознательно присутствует оверхед в виде классов и слоев абстракции. Для лендинга часть этого можно было бы упростить до нескольких функций, однако я старался писать код так, чтобы в будущем:

- легче переиспользовать модули (валидация, модалка, уведомления, наблюдатели);
- проще расширять функционал без переписывания монолита;
- понятнее зоны ответственности и границы модулей;
- удобнее поддерживать единый стиль кода и соглашения.

В общем, принципы SOLID.

## Форма, валидация и отправка

- Модальное окно открывается по клику на **Let's Talk**, закрывается по крестику и по клику на оверлей, при открытии блокируется скролл.
- Валидация формы реализована без `required` и без стандартных браузерных подсказок.
- После успешной отправки показывается уведомление, форма очищается и модалка закрывается.

Для разработки предусмотрен мок-эндпоинт `POST /contact` через dev-сервер Vite. В production окружении был настроен nginx с проксированием запросов на json-server.

## CI/CD

В репозитории настроен CI/CD пайплайн GitHub Actions (`.github/workflows/deploy.yml`).

Триггер: push в ветку `master`. Используется `concurrency` для серийных деплоев в окружение production.

Этапы пайплайна:

1. **Checkout** - получение исходников
2. **Setup Node** - Node.js 20 + кэширование npm
3. **Install** - установка зависимостей (в том числе дополнительные бинарные зависимости для линукса окружения CI, чтобы сборка была воспроизводима на runner)
4. **Lint** - проверка линтером (`npm run lint`)
5. **Build** - production-сборка (`npm run build`)
6. **Deploy** - деплой на сервер.

Параметры деплоя берутся из GitHub Secrets (`DEPLOY_HOST`, `DEPLOY_USER`, `DEPLOY_PORT`, `DEPLOY_PATH`, `DEPLOY_SSH_KEY`).

## Примечания по качеству

- Проект ориентирован на демонстрацию: семантики, адаптивной вёрстки, чистого TS, модульности и аккуратной структуры без сторонних UI/JS библиотек.
- При необходимости упрощения под реальный продакшн (если проект не будет масштабироваться и тд) большую часть оверхеда можно убрать без потери функциональности.
- Сам HTML написан "монолитом" в одном файле, потому что я не видел смысла пилить его на переиспользуемые html-блоки. Разумеется, можно было бы без проблем вынести каждый блок с фичей в отдельный модуль и тп, но насколько я знаю в реалности часто подобные лендинги хранят статичный контент, поэтому если данные изменяться не будут, то можно оставить и так. Скорее всего так будет даже лучше для SEO.
